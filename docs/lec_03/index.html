Семантика перемещения

SharedPtr

  • перемещаемый и копируемый указатель
  • используется для управления ресурсами путем совместноrо владения
  • работает на основе подсчета ссылок на объект
  • счетчик ссылок – атомарный
  • имеет два счетчик ссылок: счетчик “сильных” ссылок, счетчик “слабых” ссылок
  • блок управления находится в динамической памяти

SharedPtr

UniquePtr

  • перемещаемый и НЕ копируемый указатель
  • единалично владеет объектом
  • минимум дополнительных затрат
  • совместим с STL-контейнерами
  • поддерживает custom deleter

Проблема

String GetData(tcp::socket socket);
void Modify(String& str);
std::vector<String> storage;

for (size_t i = 0; i < N; ++i) {
  auto data = GetData(socket);
  Modify(data);
  storage[i] = data; // Копирование строки.
} // <-- Вызывается деструктор |data|.

Копирование строк

// string.h
class String {
  char* data_;
 public:
  String(const String& oth);
  String& operator=(const String& oth);
};

// string.cpp
String::String(const String& oth) {
  data_ = new char[oth.size() + 1];
  memcpy(data_, oth.data_, oth.size() + 1); // (1)
}

String& String::operator=(const String& oth) {
  if (this != &oth) {
    delete[] data_; // (2)
    data_ = new char[oth.size() + 1];
    memcpy(data_, oth.data_, oth.size() + 1); // (3)
  }
  return *this;
}

Идея 💡

String::move_from(String& b) {
  char* tmp = this->data_;
  this->data_ = b.data_;
  b.data_ = tmp;
}

for (size_t i = 0; i < N; ++i) {
  auto data = GetData(socket);
  Modify(data);
  storage[i].move_from(data); // Избавляемся от копирования.
} // <-- Вызывается деструктор пустой |data|,
  // потому что storage[i] был пустым

Move-семантина С++03

for (size_t i = 0; i < N; ++i) {
  auto data = GetData(socket);
  Modify(data);
  storage[i].swap(data); // Избавляемся от копирования.
} // <-- Вызывается деструктор пустой |data|,
  // потому что storage[i] был пустым

swap как move-семантика

Использование swap помогает повысить производительность приложения.

Однако, использовать swap для перемещения данных неудобно, а иногда и неприменимо.

Во-вторых, это усложняет и запутывает код.

Move-семантина С++11

for (size_t i = 0; i < N; ++i) {
  auto data = GetData(socket);
  Modify(data);
  storage[i] = std::move(data); // Избавляемся от копирования.
} // <-- Вызывается деструктор пустой |data|,
  // потому что storage[i] был пустым

Перемещение строк

// string.h
class String {
  char* data_;
 public:
  // Обратите внимание на noexcept!
  String(String&& oth) noexcept;
  String& operator=(String&& oth) noexcept;
};

// string.cpp
String::String(String&& oth) noexcept {
  data_ = oth.data_;
  oth.data_ = nullptr;
}

String& String::operator=(String&& oth) noexcept {
  if (this != &oth) {
    delete[] data_;
    data_ = oth.data_;
    oth.data_ = nullptr;
  }
  return *this;
}

rvalue

Rvalue ссылки – техническое расширение языка C++. Они позволяют компилятору определить, когда необходимо использовать перемещение, вместо копирования.

Перегрузка функций

// Копирование.
String(const String& oth) noexcept;
String& operator=(const String& oth) noexcept;

// Перемещение.
String(String&& oth) noexcept;
String& operator=(String&& oth) noexcept;

Благодаря rvalue, компиятор определяет какую из перегрузок функций необходимо выбрать.

std::move()

std::move безусловно приводит аргумент к rvalue ссылке.

template<typename T>
decltype(auto) move(T&& arg) {
  using ReturnType = remove_reference_t<T>&&;
  return static_cast<ReturnType>(arg);
}

std::move()

С помощью функции std::move разработчик “подсказавает” компилятору, что необходимо использовать перемещение объекта.

Рекомендации

  • Не используйте объект после того, как его переместили в другой!
Object obj;
// ...
Object another = std::move(obj);
obj.SomeMethod();

Рекомендации

  • Не объявляйте объекты константными, если хотите иметь возможность их перемещать
void foo(const Object& obj) {
  // ...
  Object another = std::move(obj); // ooops!
  // ...
}

Рекомендации

  • std::move не гарантирует, что приведенный объект будет иметь право быть перемещенным
std::array<int, 10> arr = {1, 2, 3};
// ...
std::array<int, 10> yet_arr = std::move(arr);

Резюме

  • Семантика перемещения позволяет компиляторам заменять дорогостоящие операции копирования “дешевыми” перемещениями.
  • Конструктор перемещения и оператор перемещения предоставляют контроль над семантикой перемещения.

Резюме

  • Семантика перемещения позволяет также создавать типы, которые концептуально поддерживают только операции перемещения. К таким типам относятся как std::unique_ptr, std::future, std::thread и др.
  • Это помогает передавать такие объекты в функции, возвращать их из функций, хранить в коллекциях и пр.

Шаблоны проектирования

Что это

Шаблон проектирования – повторяемая архитектурная конструкция, представляющая собой решение проблемы проектирования в рамках некоторого часто возникающего контекста.

Зачем нужны шаблоны

  • предоставляют готовое решение некоторой известной проблемы
  • позволяют поддерживать архитектуру приложения в лучшем состоянии, нежели без использования шаблонов
  • помогают разговаривать разраотчикам на одном языке
  • полезны при рефакторинге кода

Неоправданное применение

Самая большая ошибка при работе с шаблонами проектирования – это неоправданное их использование.

“Если у тебя в руках молоток, то все предметы вокруг начинают напоминать гвозди.”

Простая фабрика

В С++ существует механизм полиморфизма на базе виртуальных функций.

Полиморфизм позволяет вызывать функции производным классов, ничего не зная о них, а только оперировать базовым классом.

Полиморфизм

struct Unit {
  void Attack(Unit& target);
}

struct Knight : public Unit {
  // ...
};

struct Archer : public Unit {
  // ...
};

Unit* unit = new Knight();
Unit* target = new Archer();
unit->Attack(*target);

Виртуальный конструктор

Однако, чтобы создать объект дочернего класса необходимо явно указать в коде, объект какого класса создается.

Решить данную проблему призван шаблон проектирования “простая фабрика”

Простая фабрика

Паттерн “простая фабрика” – это класс, в котором есть один метод с большим условным оператором, выбирающим создаваемый продукт.

Этот метод вызывают с неким параметром, по которому определяется какой из продуктов нужно создать.

Реализация

struct Barrack {
  Unit* CreateUnit(UnitID unit_id) {

    switch (unit_id) {
      case UnitID::Knight:
        return new Knight();
      case UnitID::Archer:
        return new Archer();
      default:
        return nullptr;
    }

  }
};

Фабричный метод

Паттерн “фабричный метод” – это устройство классов, при котором подклассы могут переопределять тип создаваемого в суперклассе продукта.

Пример

struct Barrack {
  virtual Unit* CreateUnit() = 0;

  void Hire(Player& player) {
    auto* unit = CreateUnit();
    player.AddToArmy(unit);
  }
};

Пример (продолжение)

struct KnightBarrack : public Barrack {
  Unit* CreateUnit() override {
    return new Knight();
  }
};
struct ArcherBarrack : public Barrack {
  Unit* CreateUnit() override {
    return new Archer();
  }
};

ArcherBarrack archer_barrack;
archer_barrack.Hire(player);
KnightBarrack knight_barrack;
knight_barrack.Hire(player);

Singleton

Singleton – порождающий шаблон проектирования, гарантирующий, что в приложении будет единственный экземпляр некоторого класса, и предоставляющий глобальную точку доступа к этому экземпляру.

Назначение паттерна

Некоторые подсистемы в приложении должны существовать только в единственном экземпляре.

Например, система логгирования или драйвер дисплея.

Цели паттерна

  1. Гарантирует наличие единственного экземпляра класса

  2. Предоставляет глобальную точку доступа

Проблемы паттерна

С данным паттером традиционно связаны несколько проблем:

  • доступ и создание объекта в многопоточной среде
  • время создания и разрушения объекта singleton’a
  • взаимное использование двух синглтонов

Классическая реализация Singleton

// Singleton.h
class Singleton {
  static Singleton* instance;
  Singleton() {}
  Singleton(const Singleton&) = delete;
  Singleton& operator=(Singleton&) = delete;
public:

  static Singleton * getInstance() {
    if (!instance) {
      instance = new Singleton();
    }
    return instance;
  }
};

// Singleton.cpp
#include "Singleton.h"

Singleton* Singleton::instance = nullptr;

Реализация. Синглтон Мэйерса

// Singleton.h
class Singleton {
private:
  Singleton() {}
  Singleton(const Singleton&) = delete;
  Singleton& operator=(Singleton&) = delete;
public:
  static Singleton& getInstance() {
    static Singleton instance;
    return instance;
  }
};

Observer

Наблюдатель – это поведенческий паттерн проектирования, который создаёт механизм подписки, позволяющий одним объектам следить и реагировать на события, происходящие в других объектах.

Интерфейс Publisher

struct Publisher {
  struct Observer {
    virtual void OnUpdate() = 0;
  };
  void AddObserver(const Observer& observer);
  void RemoveObserver(const Observer& observer);
  void DoSomething();
 private:
  void Notify();
  std::vector<Observer*> observers_;
};

Реализация Publisher

void Publisher::AddObserver(const Observer& observer) {
  observers_.push_back(&observer);
}

void Publisher::RemoveObserver(const Observer& observer) {
  auto it = std::remove(
      observers_.begin(), observers_.end(), &observer);
  observers_.erase(it, observers_.end());
}

void Publisher::Notify() {
  for (auto* observer : observers_) {
    observer->OnUpdate();
  }
}

void Publisher::DoSomething() {
  // Выполнение некоторых действий.
  // Решаем, что необходимо умедомить наблюдателей о событии.
  Notify();
  // Продолжчаем выполнение функции.
}

Использование Observer

struct SubscriberOne : public Publisher::Observer {
  void OnUpdate() {
    // Реагирование на уведомление от Publisher'a
  }
};
struct SubscriberTwo : public Publisher::Observer {
  void OnUpdate() {
    // Реагирование на уведомление от Publisher'a
  }
};

Publisher publisher;
SubscriberOne subscriber1;
SubscriberTwo subscriber2;
publisher.AddObserver(subscriber1);
publisher.AddObserver(subscriber2);
publisher.DoSomething();
